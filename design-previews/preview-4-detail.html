ACM-119 billing P1 · Created May 21 · Updated 12m ago

Stripe webhook retry with idempotency keys

Webhooks occasionally double-fire when our handler returns 500 — Stripe retries aggressively, and we end up applying the same plan change twice. Symptoms surface as duplicate audit log entries and (rarely) double-charges on plan upgrades.

Proposed fix

  • Wrap the webhook handler in a Redis-backed dedup window of 60s, keyed by event.id.
  • Return 200 OK immediately for any event we've seen in the last 60s.
  • Surface the dedup hit-rate in Grafana so we can verify it's working in prod.

Notes

Spoke to @julia about this — she pointed out that Stripe's docs already recommend this pattern but we never implemented it. See the screenshot below for our current retry rate.
